# QA Tester (James Whittaker) — Learnings

Patterns of bugs found, missed edge cases, and testing blind spots.
Read this file before starting any QA session.

## Rules

<!-- Entries are added automatically by retro. Format:

- [2026-03-18] When testing documentation, always cross-check requirements.txt comment text against actual listed packages — comments can claim a package is listed when it isn't. (source: James Whittaker)
- [2026-03-18] When testing documentation prose that counts items ("Two ways"), verify the count matches the actual bullet points that follow. (source: James Whittaker)
- [2026-03-18] When docs use shell examples (base64, curl flags), verify portability across macOS vs Linux — macOS-specific flags like `base64 -i` are not portable. (source: James Whittaker)
- [2026-03-18] Always check every endpoint in the API reference has an "Error Conditions" section if it can 500 on non-primary platforms; compare against sibling endpoints for consistency. (source: James Whittaker)
- [2026-03-18] When testing dynamically-created UI slots (character cards, list items), verify that index variables captured in closures at creation time are still valid after items are removed or reordered. Stale idx bugs cause wrong labels and potentially wrong data targets. (source: James Whittaker)
- [2026-03-18] When testing multi-value input fields, run the exact string transformation chain from the code against the placeholder example text — the placeholder may promise comma-separated input that the code does not actually support. (source: James Whittaker)
- [2026-03-18] When frontend reads a backend JSON response, always verify the exact key name matches — e.g., backend returns {"characters": [...]} but frontend reads data.tags will silently return undefined, not an error. Check every data.* access against the actual response shape. (source: James Whittaker)
- [2026-03-19] When a route handler performs pre-processing before the main try/except block (e.g., image compositing before calling an external API), test it with malformed input — exceptions thrown in the pre-processing step will produce a 500, not the expected 502. The pre-processing must be inside or wrapped by the same try/except. (source: James Whittaker)
- [2026-03-19] When testing "restore/load" UI flows, always test against the full dataset distribution — not just the happy-path case. If 92% of records lack a field, a restore function that unconditionally clears state before checking for that field will silently destroy user data on the majority of invocations. Always verify the clear is conditional on the presence of replacement data. (source: James Whittaker)
- [2026-03-19] When a function parses structured sub-data (e.g. interaction directives stripped from a prompt string) and stores it in a data object, verify the UI-building function that consumes that object also renders the pre-existing sub-data — not just provides a way to add new entries. Parsed data that is stored but never rendered is invisible to the user. (source: James Whittaker)
- [2026-03-20] When testing features that involve ResizeObserver or MutationObserver, verify the observer callback does not have destructive side-effects (like clearing canvas content) that fire on every observation cycle. Observers fire frequently and asynchronously — any state-clearing code inside them will silently wipe user work. (source: James Whittaker)
- [2026-03-20] When testing dblclick interactions on elements that also handle single-click with re-render, verify the dblclick still fires after the first click's handler runs. If the first click destroys and recreates the DOM element, the browser cannot fire dblclick on the new element. (source: James Whittaker)
- [2026-03-20] When testing loadSettingsFromMeta across the full image dataset, verify every DOM element accessed is null-checked. Elements may not exist due to hidden features (SMEA), removed features, or responsive breakpoints. A single unchecked null access crashes the entire function silently. (source: James Whittaker)
- [2026-03-20] When a frontend render function receives backend data, verify not just the top-level key names but also the exact field names inside each item. A flat array of {name, score} objects processed through Object.entries() produces numeric string keys as "categories"; destructuring {tag, score} from {name, score} yields undefined and causes TypeError: tags is not iterable — total feature failure with no obvious error. (source: James Whittaker)
- [2026-03-20] When a JS function expects a grouped dict (Object.entries iteration) but receives a flat array, the TypeError "tags is not iterable" is caught silently and displayed as a user-visible error, not a developer console error. Always trace data shape from backend schema through response parsing to the render function that consumes it. (source: James Whittaker)
- [2026-03-20] When testing tag insert/remove toggling, verify the remove function handles the solo-tag case (tag with no surrounding commas). Regex patterns that require a leading comma OR a trailing comma both fail on a prompt containing only that one tag. (source: James Whittaker)
- [2026-03-21] When testing multi-provider features, cross-check every enum value in the backend schema's Literal type against every <option> in the corresponding HTML <select>. A backend that accepts "2:1" and "1:2" while the HTML only shows 7 of 9 options silently truncates valid user choices without error. (source: James Whittaker)
- [2026-03-21] When a `finally` block sets a UI label before calling a shared reset function, verify the reset function does not immediately overwrite the label. Always check the execution order: if resetGenerateButton() unconditionally writes "Generate", any label set before it in the same finally block is silently clobbered. (source: James Whittaker)
- [2026-03-21] When a config bar hides provider-specific sections on provider switch, audit every separator element independently. A separator between two always-visible elements (e.g. Provider and Canvas) will become a dangling orphan if one of the elements it separates is hidden — it must be hidden together with the element it precedes. (source: James Whittaker)
- [2026-03-22] When a CSS class is defined twice in the same stylesheet (once for new behavior, once labeled "legacy"), always verify which definition comes last — the later rule wins at equal specificity. A "legacy/compat" rule with `display: none` placed after the real rule will silently hide the element. Grep for duplicate selectors whenever refactoring styles. (source: James Whittaker)
- [2026-03-22] When a shared keydown handler (e.g. tag autocomplete) handles Escape by calling hide() without stopPropagation(), any parent/document-level Escape handler will also fire. In a modal containing a dropdown, this means pressing Escape to dismiss the dropdown will simultaneously close the entire modal. Always call e.stopPropagation() when Escape is consumed by a child handler inside a modal. (source: James Whittaker)
- [2026-03-22] When a CSS animation is applied to a persistent DOM element that is shown/hidden via display toggling (not created/destroyed), the animation only plays once — on first show. Subsequent shows skip the animation because the element has already completed it. Use a JS reflow trick (el.offsetWidth; el.style.animation = "none"; requestAnimationFrame(() => el.style.animation = "")) to force replay on re-open. (source: James Whittaker)
