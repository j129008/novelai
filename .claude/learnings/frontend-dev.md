# Frontend Dev (Lea Verou) — Learnings

Rules and patterns learned from code reviews, QA feedback, and PM feedback.
Read this file before starting any implementation task.

## Rules

<!-- Entries are added automatically by reviewers and retro. Format:
- [YYYY-MM-DD] <rule> (source: <reviewer>) -->
- [2026-03-18] Do not expose internal functions via window.* unless there is a concrete external caller. Dead global assignments are code smell. (source: Linus Torvalds)
- [2026-03-18] When adding a sticky footer to a scrollable container, use flex column layout (parent flex-direction:column, scroll area flex:1 overflow-y:auto, footer flex-shrink:0) — not position:sticky which can fail in nested scroll contexts. (source: Linus Torvalds, confirmed by PM)
- [2026-03-18] Accordion empty states must never be visually empty. When content is dynamically added (e.g. character slots), provide a visible empty state card with a clear call-to-action so users know the feature is working and what to do next. A blank accordion body = perceived broken feature. (source: Steve Jobs)
- [2026-03-18] Abstract grid pickers (e.g. position selectors) require axis context. Never render a bare NxN grid without labels, directional cues, or a shaped border that maps to the underlying coordinate space. Without context, users click randomly and blame the output. (source: Steve Jobs)
- [2026-03-18] Do not place dynamic UI chips/hints outside the accordion that triggered them. If a chip (e.g. "Add 2girls to prompt") only appears when the Characters accordion is in use, it must live inside that accordion — not in the main sidebar flow where users won't see it. (source: Steve Jobs)
- [2026-03-18] Hoist loop-invariant constants above the loop. Do not recompute fixed values (e.g. Math.floor(ROWS/2)) inside nested loops where they execute N×M times. (source: Linus Torvalds)
- [2026-03-18] Never put overflow:hidden on a container that has position:absolute children expected to overflow it (e.g. autocomplete dropdowns, tooltips). overflow:hidden clips absolutely-positioned descendants that extend past the box edge. Use max-height on the specific element that needs capping, not the parent. (source: Linus Torvalds)
- [2026-03-18] When dynamically-created list items (character cards, slots) capture their own index at creation time, that index becomes stale after earlier items are removed. Either refresh all captured indices on remove, or derive the current index from DOM position at the time of use. (source: James Whittaker)
- [2026-03-18] Placeholder text for an input field must exactly match what the code accepts. If the transformation chain only supports a single tag per input, the placeholder must not show a comma-separated list — it misleads users into producing malformed data. (source: James Whittaker)
- [2026-03-18] Never interpolate server-sourced metadata (seed, steps, dimensions, filenames) directly into innerHTML. Even in local apps this is wrong. Use createElement+textContent or Number() conversion with type verification. (source: Linus Torvalds)
- [2026-03-18] Do not rely on a default value coincidence to signal "no preference" to the backend. If a user picks grid cell (center, center) which happens to be x=0.5,y=0.5 — the same as the auto default — the backend silently ignores their explicit choice. Use an explicit boolean flag or conditional payload key to distinguish "user chose center" from "auto". (source: Linus Torvalds)
- [2026-03-18] When reading a fetch() response, always use the exact key the server returns. data.tags when the server sends {"characters": [...]} silently returns undefined; use data.characters. Cross-check every data.* access against the backend route's response_model or return dict. (source: James Whittaker)
- [2026-03-18] Sidebar control order must reflect actual usage frequency, not feature enumeration. Controls touched every generation (Characters, img2img) go above the fold; controls touched rarely (Canvas Size, Seed, Steps, Sampler) must be collapsed or compacted. Treating all controls as equal importance forces the user to hunt for critical controls past irrelevant ones. (source: Steve Jobs)
- [2026-03-18] A history/gallery view that sends the user to a different page to view a single image breaks the browsing flow. The "appreciate" use case requires a lightbox that overlays the current page with keyboard navigation — never navigate away from the gallery to preview a single item. (source: Steve Jobs)
- [2026-03-18] Recall and discovery are different cognitive modes and must not share the same UI surface. Recent/used items belong near the point of re-use (e.g. recent characters inside the Characters accordion). Discovery tools (Tag Browser) must stay clean of personal history to maintain the "try something new" mindset. (source: Steve Jobs)
- [2026-03-18] Gallery cards at minmax(160px, 1fr) produce 4-5 columns that are too small to appreciate image quality. For image galleries where the primary use is visual evaluation, use minmax(240px, 1fr) as the minimum — never go below this for AI-generated image content. (source: Steve Jobs)
- [2026-03-18] Combine rarely-changed controls (Canvas Size + Seed) into a single compact two-column row rather than giving each a full control-group block. Every vertical pixel in a sidebar is precious real estate; constant controls must not compete with frequent controls for vertical space. (source: Steve Jobs)
- [2026-03-19] Canvas overlays (draggable markers, annotations) must only render when there is actual canvas content to overlay. Rendering markers over a placeholder state creates confusion — the user has no image to reason about, so coordinates are meaningless. Gate all canvas overlays on the existence of a real image. (source: Steve Jobs)
- [2026-03-19] Never hide a non-obvious interaction (double-click to reset, long-press, etc.) without a visible affordance. At minimum, add a title tooltip. Without it, users who accidentally trigger a drag have no recovery path and no mental model of the feature. (source: Steve Jobs)
- [2026-03-19] Touch and mouse event handlers that diverge in side-effects will produce inconsistent UX across device types. If a mouse click highlights a sidebar card, the equivalent touch tap must do the same. Audit every touch/mouse handler pair for behavioral parity. (source: Steve Jobs)
- [2026-03-19] Never use hardcoded color values (white, black, #fff) in component CSS — always use design system tokens (--text-primary, --bg-deep, etc.). Hardcoded colors break theme integrity and are invisible in code review without a token audit. (source: Steve Jobs)
- [2026-03-19] Document-level event listeners (mousemove, mouseup) registered inside a drag-start handler must be cleaned up if the dragged element is destroyed before mouseup fires. If a re-render can destroy the element mid-drag, use a module-level drag state that the re-render function resets, so the document listener checks a live flag rather than referencing stale DOM. (source: Linus Torvalds)
- [2026-03-19] Do not update DOM class state inside drag-start handlers to reflect data state that has not yet changed. Visual state must reflect data state. If data only changes at drag-end (conditionally), the class must only change at drag-end. Premature class removal on mousedown causes visible flicker when the interaction turns out to be a click, not a drag. (source: Linus Torvalds)
- [2026-03-19] Comments that describe behavior the code does not implement are actively harmful — they make reviewers trust the code is complete when it is not. Either implement the described behavior or delete the comment. Zero tolerance for comment/code drift. (source: Linus Torvalds)
- [2026-03-19] Never unconditionally clear shared state (arrays, DOM containers) at the start of a restore/load operation. Always check whether the incoming data actually has replacement content first. Unconditional clears silently destroy user data when the source has no replacement — e.g., clearing character slots when loading settings from an image that was generated without characters. (source: James Whittaker)
- [2026-03-19] When a factory function (e.g. buildInteractionsSection) creates a UI widget for a data object that may already have pre-existing sub-data (e.g. charData.interactions), it must render those existing entries on creation — not just provide an add-new UI. Failing to pre-populate makes restored data invisible to the user. (source: James Whittaker)
- [2026-03-20] A sidebar is not a settings panel. It is a creative tool. Controls that are touched every generation must be immediately visible without scrolling. Controls that are configured once per session must be removed from the main sidebar flow entirely — use a popover, modal, or drawer — not another accordion in the same scroll column. Repeated user complaints about sidebar crowding signal a structural failure, not a cosmetic one. (source: Steve Jobs)
- [2026-03-20] When a sidebar has more than 3–4 control groups, stop adding compact rows and start asking what does not belong in a sidebar at all. The answer is almost always "everything except the primary creative input and the thing the user changes most often." Configuration belongs in a popover triggered by a single compact row. (source: Steve Jobs)
- [2026-03-20] A popover for secondary controls should be anchored and constrained — position:absolute inside the parent sidebar container, never position:fixed. Fixed popovers escape their parent and create z-index and clipping bugs across breakpoints. (source: Steve Jobs)
- [2026-03-20] Setting canvas.width or canvas.height clears ALL content (HTML5 spec). If a ResizeObserver or resize handler sets dimensions, guard with `if (canvas.width === newW && canvas.height === newH) return` — otherwise every observer tick erases user-drawn strokes. (source: David, user bug report)
- [2026-03-20] Never call a full re-render (destroy + recreate DOM) inside a single-click handler if the element supports dblclick. The second click lands on a new element and the browser cannot fire dblclick. Update classes in-place on single click; only re-render on state changes that alter the element count. (source: David, user bug report)
- [2026-03-20] When loadSettingsFromMeta accesses optional DOM elements (hidden controls, removed features), every access must be null-guarded. A single null.checked TypeError crashes the entire function and prevents all subsequent code (like character restoration) from running. (source: David, console error)
- [2026-03-20] When a "Load" action restores settings from a secondary view (History, Lightbox), always auto-switch to the primary working view (Canvas tab) afterward. Leaving the user on a non-actionable tab after loading forces unnecessary manual navigation. (source: David, user feedback)
- [2026-03-20] When adding automatic behavior that replaces a manual action (e.g., auto-insert image at cursor replaces Insert Image button), immediately remove the now-redundant manual control. Leaving dead buttons creates confusion about which is the "real" way to do things. (source: David, user feedback)
- [2026-03-20] When a render function iterates a backend response with Object.entries(), the parameter must be an object keyed by category, not a flat array. If the API returns a flat list, group it by a field (e.g. category) client-side before passing to the renderer — or change the renderer to iterate the array directly. Never pass a flat array to a dict-iteration renderer. (source: James Whittaker)
- [2026-03-20] When reading fields from backend response objects, use the exact field name from the response_model schema. item.tag is undefined if the schema declares name — use item.name. Always cross-check every object property access against the Pydantic schema's field names. (source: James Whittaker)
- [2026-03-20] removeTagFromPrompt must handle the solo-tag case: a prompt containing exactly the tag with no commas. Both the leading-comma pattern and the trailing-comma pattern miss this case. Add a fallback: if the cleaned value still equals the original tag, set value to empty string. (source: James Whittaker)
- [2026-03-20] When removing large code blocks, delete them entirely — do not wrap in dead-function wrappers or block comments. Commented-out code with mismatched braces causes syntax errors; dead functions with `return` at the top still occupy scope and confuse IDEs. (source: Linus Torvalds, syntax error)
- [2026-03-20] contentEditable with embedded non-editable blocks (figures, cards) is inherently fragile. Use MutationObserver to re-enforce contentEditable=false on protected elements, event delegation for action buttons (they survive DOM mutations), and pointer-events:none on images to prevent browser resize handles. (source: David, user bug report)
- [2026-03-20] When building a feature that has multiple card-building code paths (new, cache restore, load from metadata), either extract a shared builder function or ensure every code path includes all interactive elements (reorder buttons, delete buttons, etc.). Missing buttons on cached/restored items is a recurring bug. (source: David, repeated bug reports)
